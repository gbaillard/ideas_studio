## Resume executif

Le marche des jeux educatifs enfants est en pleine explosion : **$38B en 2026, projete a $327B en 2035** (CAGR 27%). Le marche des apps familiales par abonnement vaut **$7.2B en 2024** (CAGR 12.4%, vers $20.4B en 2033). L'App Store a genere **$52.5B de revenus gaming en 2025**.

**Modele recommande pour PERIPLO** : Freemium + abonnement familial ($4.99/mois ou $39.99/an), avec rewarded ads optionnelles dans la version gratuite. Le hard paywall convertit 5x mieux (10.7% vs 2.1%) mais limite la base utilisateurs. La strategie optimale est **freemium avec trial 7 jours** vers abonnement.

**Chiffre cle** : 55% des annulations de trial de 3 jours se font au Jour 0 -- la valeur doit etre demontree immediatement.

- Ajout de **rewarded ads** dans la version gratuite (entre les niveaux, 1 par session max)

- **Pack ponctuel** : themes premium a $1.99-$2.99 (pour ceux qui ne veulent pas d'abo)

#### Phase 3 : Scale (Annee 2+)

- **Partenariats ecoles** (licence B2B, $2-$5/eleve/an)
- **API de contenu** pour integration avec d'autres plateformes
- **Merchandising** si base users > 1M

## PARTIE 3 -- ERREURS A EVITER

---

### 3.1 Erreurs de monetisation

1. **Monetisation agressive ciblant les enfants** -- Prodigy a ete critique pour des tactics "pay-to-win" visant les enfants. Les dark patterns (loot boxes, virtual currencies, FOMO) sont toxiques pour la reputation et risquent des poursuites FTC.

2. **Consumables et virtual currencies** -- Le modele "monnaie virtuelle" fonctionne mal pour les apps enfants : resistance parentale forte, regulations strictes (age gates, etapes d'achat supplementaires). Privilegier les achats permanents (themes, packs de contenu).

3. **Interstitial ads non consenties** -- Les pubs plein ecran entre chaque action detruisent l'experience et generent des reviews 1 etoile. Les parents desinstallent immediatement.

4. **Ignorer le "two-customer problem"** -- L'utilisateur (enfant) != le client (parent). L'enfant doit trouver le jeu fun, le parent doit percevoir la valeur educative et la securite. Les deux doivent etre satisfaits.

5. **Prix trop bas** -- Se brader a $0.99/mois devalue le produit et rend impossible la croissance. Les parents qui investissent dans l'education de leur enfant acceptent $4.99-$9.99/mois si la valeur est demontree.

### 3.2 Erreurs de design

6. **Traiter tous les enfants comme un groupe homogene** -- Un enfant de 3 ans et un de 9 ans ont des besoins radicalement differents. Concevoir par tranche d'age (3-5, 5-7, 7-10) avec des UX adaptees.

7. **Menus et navigation complexes** -- Les jeunes enfants ne savent pas lire et ont une motricite fine limitee. Boutons enormes, navigation par icones, instructions audio.

8. **Pas de test avec de vrais enfants** -- Les adultes ne peuvent pas predire comment un enfant de 4 ans interagira avec une interface. Tester avec de vrais enfants et leurs parents est indispensable.

### 3.3 Erreurs de compliance

9. **SDK analytics/tracking dans une app Kids** -- Apple rejette les apps Kids Category qui incluent des SDK collectant des identifiants (IDFA, device ID). Nettoyer TOUS les SDK avant soumission.

10. **Age gate naive** -- Demander "as-tu 13 ans ?" ne suffit pas si l'app est clairement destinee aux enfants. La FTC considere cela comme une violation.

11. **Ignorer les SDK privacy manifests (2025+)** -- Apple exige desormais des manifestes de confidentialite et des signatures pour tous les SDK tiers. Les apps non conformes sont rejetees.

### 3.4 Erreurs de lancement

12. **Croire que l'app se vendra seule** -- Meme une app excellente disparait parmi les millions d'apps existantes sans strategie de marketing active.

13. **Lancer globalement sans soft launch** -- Ne pas tester dans 1-3 petits marches avant le lancement mondial. Les metriques de soft launch (retention, conversion, crash rate) sont essentielles.

14. **Pas de strategie de reviews** -- Les reviews sont le facteur #1 de conversion dans l'App Store. Ne pas demander de reviews au bon moment = invisibilite.

## PARTIE 4 -- STRATEGIE DE LANCEMENT OPTIMALE

---

### Phase Pre-Lancement (J-60 a J-0)

**Semaine 1-4 : Preparation**
- [ ] Creer la page App Store / Google Play avec ASO optimise (voir Partie 5)
- [ ] Lancer un site web simple (landing page + email capture)
- [ ] Creer les comptes reseaux sociaux (Instagram, TikTok, Pinterest -- ou les parents sont)
- [ ] Preparer les assets marketing : 30-60s trailer, screenshots, press kit
- [ ] Identifier 20-50 micro-influenceurs parents/education (1K-50K followers)

**Semaine 5-8 : Beta & Community**
- [ ] Beta privee avec 50-100 familles (mix enfants 3-10 ans)
- [ ] Collecter feedback et iterer sur UX/bugs
- [ ] Open beta via Google Play (pas de reviews publiques pendant beta)
- [ ] Documenter les reactions des enfants en video (avec accord parental) pour le marketing
- [ ] Contacter les influenceurs avec des acces gratuits

### Phase Soft Launch (J-0 a J+30)

**Marches de test recommandes :**
- **iOS** : Canada, Australie, Nouvelle-Zelande (marches anglophones, plus petits que US)
- **Android** : Pays-Bas, Belgique, Suede (marches europees, tech-savvy)

**Metriques a tracker :**
| Metrique | Seuil minimum | Objectif |
|----------|--------------|----------|
| D1 retention | > 35% | > 45% |
| D7 retention | > 18% | > 28% |
| Crash rate | < 1% | < 0.5% |
| Session length | > 5 min | > 8 min |
| Trial start rate | > 15% | > 25% |
| Trial-to-paid | > 3% | > 5% |

### Phase Hard Launch (J+30 a J+90)

- [ ] Lancement US + principaux marches EU (FR, DE, UK, ES, IT)
- [ ] Campagne PR : pitch aux blogs parents, sites education, media tech
- [ ] Activation micro-influenceurs (envoi coordonne la semaine du launch)
- [ ] Reddit : posts sur r/parenting, r/homeschool, r/educationalgames (valeur d'abord, pas de spam)
- [ ] Demander des reviews in-app apres 3 sessions completees (pas avant)
- [ ] Feature request Apple : soumettre via App Store Connect 6+ semaines avant le lancement

### Phase Post-Launch (J+90+)

- [ ] Publier un nouveau jeu/theme toutes les 2-4 semaines (retention)
- [ ] Email nurture pour les users gratuits (mise en avant des features premium)
- [ ] A/B test des paywalls (timing, message, prix)
- [ ] Iterer sur ASO chaque mois (keywords, screenshots, description)
- [ ] Commencer les partenariats ecoles/associations de parents

## PARTIE 5 -- ASO (App Store Optimization) POUR JEUX ENFANTS

---

### 5.1 Differences iOS vs Google Play

| Element | iOS App Store | Google Play |
|---------|--------------|-------------|
| **Champs indexes** | Title, Subtitle, Keywords field | Title, Short Description, Full Description |
| **Titre** | 30 caracteres max | 30 caracteres max |
| **Sous-titre/short desc** | 30 car (subtitle) | 80 car (short description) |
| **Keywords** | Champ dedie 100 car (invisible) | Integres dans description (~1 match exact / 250 car) |
| **Algorithme** | Qualite d'engagement (28-day playtime/user) | Engagement + "Level Up" program (2025+) |
| **Nouveaute 2025** | **App Store Tags** (IA genere des labels depuis metadata + screenshots) | Engagement-first ranking |

### 5.2 Keywords strategy pour PERIPLO

**Keywords primaires (haute intention) :**
- jeux educatifs enfants, educational games kids, learning games, jeux apprentissage

**Keywords secondaires (volume) :**
- jeux 3 ans, jeux 5 ans, jeux maternelle, preschool games, kids puzzle, memory game kids

**Keywords long-tail (faible competition) :**
- jeux logique enfants, pattern game kids, odd one out kids, jeux voyage enfants

**Keywords parents (decision d'achat) :**
- safe kids app, no ads kids game, educational screen time, COPPA compliant

### 5.3 Screenshots qui convertissent

**Regles pour apps enfants :**
1. **Screenshot 1** : Core value prop en < 2 secondes ("Jeux educatifs sans pub pour 3-10 ans")
2. **Screenshot 2** : Enfant jouant, couleurs vives, interface visible
3. **Screenshot 3** : Ce que l'enfant apprend (maths, logique, memoire)
4. **90% des users ne scrollent pas apres le screenshot 3** -- tout doit etre dit en 3 images
5. Captions courtes : 5-8 mots par screenshot, focus benefice (pas feature)
6. Video preview : 15-30 secondes de gameplay reel

**Pour les parents specifiquement :**
- Mettre en avant "Sans pub" / "Sans achats surprises"
- Montrer le rapport de progres parental
- Mentionner les tranches d'age clairement
- Badge "Education" ou "Approuve par des pedagogues" si applicable

**Localisation** : Les apps qui localisent screenshots pour les 5 premiers marches voient **+25-30% de telechargements** dans ces regions.

### 5.4 Optimisation continue

- Iterer les keywords **chaque mois** (suivre rankings avec AppTweak, ASOMobile, ou MobileAction)
- A/B tester les screenshots (Apple propose des Custom Product Pages)
- Saisonnier : adapter visuels pour rentree scolaire (Sept), Noel (Nov-Dec), ete (Juin)
- Surveiller les **App Store Tags** generes par l'IA d'Apple -- ils influencent la decouverte

> Source : [CAS.AI ASO 2026](https://cas.ai/blog/aso-2026-app-store-optimization/), [ASOMobile Complete Guide 2026](https://asomobile.net/en/blog/aso-in-2026-the-complete-guide-to-app-optimization/), [Appalize ASO for Games](https://www.appalize.com/blog/aso-strategies/aso-for-games-how-to-optimize-your-mobile-game-for-the-app-stores)

## PARTIE 6 -- COMPLIANCE COPPA / GDPR / KIDS CATEGORY

---

### 6.1 COPPA (USA) -- Mise a jour 2025

**Qui est concerne** : Toute app "dirigee vers les enfants" ou qui a "connaissance reelle" que des enfants < 13 ans l'utilisent.

**Obligations principales :**
1. **Consentement parental verifiable** avant toute collecte de donnees personnelles
2. **Politique de confidentialite claire** expliquant : quoi est collecte, pourquoi, ou c'est stocke, avec qui c'est partage, combien de temps c'est conserve
3. **Data map obligatoire** : la FTC attend un inventaire complet des donnees
4. **Identifiants persistants** (device IDs, cookies, IDFA) = donnees personnelles sous COPPA
5. **Mesures de securite raisonnables** pour proteger les donnees

**Mise a jour 2025 :**
- Definition elargie des "donnees personnelles" (inclut biometrie, photos, voix)
- Exigences renforcees sur les SDK tiers ("child-suitable" uniquement)
- Penalites accrues pour non-conformite

### 6.2 GDPR-K (Europe)

- Consentement parental obligatoire pour le traitement des donnees des mineurs
- **Privacy by default** : parametres de confidentialite au maximum par defaut
- Pas de profilage comportemental des enfants
- Droit a l'effacement renforce

### 6.3 UK Children's Code (Age Appropriate Design Code)

**15 standards obligatoires** dont :
- **Best interests of the child** : l'interet de l'enfant prime dans toutes les decisions design
- **Data minimisation** : collecter le strict minimum
- **High privacy by default** : pas de partage, pas de tracking par defaut
- **No nudging/dark patterns** : interdit de pousser l'enfant a baisser sa protection
- **Depuis juin 2025** : integre dans UK GDPR Article 25(1) = obligation legale contraignante

### 6.4 Apple Kids Category -- Restrictions specifiques

**Interdit :**
- SDK analytics tiers collectant IDFA ou identifiants
- SDK publicitaires avec tracking comportemental
- Envoi de donnees personnelles ou device info a des tiers
- Links externes sortant de l'app (sauf parental gate)

**Autorise avec restrictions :**
- Analytics tiers SI zero collecte d'identifiants (anonyme seulement)
- Publicite contextuelle SI le reseau a des pratiques documentees pour Kids Category + review humain des creatives

**Nouveaute 2025-2026 :**
- **SDK Privacy Manifests** obligatoires pour tous les SDK tiers
- **Signatures SDK** requises (supply chain integrity)
- Mise a jour des age ratings obligatoire avant **31 janvier 2026**
- Nouvelles categories d'age : 13+, 16+, 18+
- **Divulgation obligatoire** de tout partage de donnees avec des IA tierces

### 6.5 Checklist compliance PERIPLO

```
[ ] Zero collecte de donnees personnelles des enfants
[ ] Zero SDK tracking (pas de Firebase Analytics classique, pas de Facebook SDK)
[ ] Analytics anonymes uniquement (ex: Mixpanel anonyme, custom analytics)
[ ] Pub via Kidoz ou SuperAwesome uniquement (si pub)
[ ] Parental gate pour tout achat in-app
[ ] Parental gate pour tout lien sortant
[ ] Politique de confidentialite claire et accessible
[ ] Data map documentee (quoi/pourquoi/ou/qui/combien de temps)
[ ] Privacy manifest pour chaque SDK tiers
[ ] Age rating correctement configure (mis a jour avant Jan 2026)
[ ] Test compliance avant chaque soumission App Store
[ ] Pas de tracking, cookies, ou identifiants persistants cote enfant
```

## PARTIE 8 -- PLAN D'ACTION PERIPLO

---

### Priorite immediate (Semaine 1-2)

1. **Definir le modele freemium** : quels 3 jeux sont gratuits, quels contenus sont premium
2. **Implementer le parental gate** pour tous les achats et liens sortants
3. **Auditer tous les SDK** : supprimer tout ce qui collecte des identifiants
4. **Preparer la page App Store** avec ASO optimise (keywords, screenshots, video)
5. **Implementer RevenueCat** pour la gestion des abonnements (gratuit jusqu'a $2.5K MRR)

### Priorite court terme (Semaine 3-6)

6. **Beta privee** avec 50+ familles
7. **Integrer Kidoz ou SuperAwesome** pour les ads (si modele avec pub)
8. **Creer le site web** + landing page avec email capture
9. **Contacter 20+ micro-influenceurs** parents/education
10. **Preparer le press kit** (screenshots HD, video 30s, description, logos)

### Priorite moyen terme (Mois 2-3)

11. **Soft launch** dans 2-3 marches test
12. **Iterer** sur conversion trial-to-paid et retention
13. **Hard launch** US + EU avec campagne coordonnee
14. **Soumettre pour feature Apple** (Education, Kids)
15. **Premiers partenariats** ecoles/enseignants
